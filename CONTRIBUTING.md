 PrivacyGuard 贡献指南
欢迎参与本隐私防护工具的功能开发、漏洞修复、文档完善与版本迭代，所有贡献者请遵守本文档协作规范。

 一、分支管理规范
1. main 分支：稳定发布分支，禁止直接 push 代码，仅合并审核通过的 PR
2. feature/xxx：新功能开发分支，如 feature/batch-encrypt
3. fix/xxx：bug、隐私安全漏洞修复分支，如 fix/memory-leak
4. docs/xxx：README、说明文档修改专用分支

本地开发流程：拉取 main 最新代码 → 新建独立分支 → 开发自测 → 提交 PR

 二、本地开发环境准备
1. 克隆仓库到本地
2. 安装依赖
```bash
pip install cryptography
提交代码前本地运行全部单元测试，保证加解密功能正常、无隐私泄露风险

Issue 使用规范
新增功能、安全漏洞、优化建议都需要先创建 Issue
打标签区分类型：feature /bug/docs /security
安全漏洞类 Issue 写明复现步骤、数据泄露风险、修复思路
需要认领任务在 Issue 评论区留言，避免多人重复开发同一功能

四、隐私安全强制编码规则
本项目为隐私加密工具，安全要求优先于功能：
禁止硬编码密钥、明文隐私数据
程序退出自动清空内存明文缓存，防止数据残留
加密统一使用 AES256，不允许简易弱加密算法
文件路径增加合法性校验，防止越权访问本地文件
新增功能必须配套单元测试，校验加密完整性、错误密钥拦截逻辑

五、Pull Request 提交流程
PR 必须关联对应 Issue 编号
PR 描述写明改动内容、测试结果、解决的隐私安全问题
至少一名其他成员完成代码 Review，重点排查数据泄露隐患
根据 Review 意见修改代码后重新推送等待复审
GitHub Actions 自动化安全扫描、单元测试全部通过方可合并

六、代码 Review 评审标准
可用性：加解密流程完整，无崩溃、无异常报错
安全性：无明文缓存、无密钥泄露、不存在弱加密逻辑
代码规范：命名清晰，关键逻辑添加注释，无冗余无效代码
测试覆盖：新增业务逻辑配套对应测试用例

七、Git Commit 提交格式
统一格式：[类型] 简短描述
[Feature] 新增隐私防护功能
[Fix] 修复 bug 或隐私安全漏洞
[Docs] 更新项目文档
[Refactor] 代码重构，不修改功能逻辑
示例：
[Fix] 修复运行后内存明文残留泄露问题

八、版本发布规则
仅项目组长负责打 Tag 和发布 Release，发布前需满足：
全部核心功能可正常离线运行
自动化安全检测、单元测试无告警
完善 Release Notes，记录新增功能与修复的安全问题
